# Repository Guidelines

## Project Structure & Module Organization
- `apps/` holds deployable applications (frontend Angular, backend Effect-TS). Each app typically contains `src/`, config, and project-specific tests.
- `libs/` contains shared libraries used across apps; keep APIs stable and avoid cross-boundary imports that violate Nx module constraints.
- `tools/` contains shared build/deploy tooling (including Nx executors).
- `docs/` stores longer-form documentation.
- `dist/` is build output (generated by Nx); don’t edit by hand.

## Build, Test, and Development Commands
- `pnpm run lint:affected` lints all affected projects.
- `pnpm run test:affected` runs affected unit tests.
- `pnpm run build:affected` builds affected projects (and storybook where configured).
- Use `pnpm exec nx <target> <project>` for project-scoped tasks, e.g., `pnpm exec nx test user-flow-manager`.

## CI/CD Execution Policy
- All CI/CD operations must run via Nx targets (`pnpm exec nx ...`), including deployment workflows.
- CI/CD deployment/release logic must be implemented as Nx executors (for example under `tools/`), not standalone scripts.
- Do not use `nx:run-commands` + script wrappers for CI/CD deployment/release targets.
- Do not invoke raw `node`, `docker`, or `aws` commands directly in CI/CD jobs; call the appropriate Nx executor-backed target instead.

## Coding Style & Naming Conventions
- Indentation: 2 spaces; max line length 120.
- Prettier is the source of truth (`.prettierrc`); use single quotes and trailing commas.
- ESLint enforces Nx module boundaries; avoid importing across disallowed library boundaries.
- SCSS is the default style language for Angular generators.

## Testing Guidelines
- Unit tests use Vitest (Nx + Vite setup).
- E2E tests are set up with Cypress (run via Nx targets).
- Name tests `*.spec.ts`/`*.test.ts` and keep them near the source when practical.
- Run CI-style coverage with `pnpm exec nx test <project> --configuration=ci` (when the target is configured).

## Commit & Pull Request Guidelines
- Commit messages must follow Conventional Commits (Angular preset), e.g., `feat(ui): add audit status badge`.
- Husky/commitlint will block non-compliant commits.
- PRs should include a clear summary, testing notes (what you ran), and screenshots for UI changes.
- CI runs lint/test/build on PRs; keep changes scoped so `nx affected` remains fast.

## Security & Configuration Tips
- Keep secrets out of the repo; use environment variables and local `.env` overrides.
- Deployment and CI are managed via GitHub Actions; avoid direct edits to `dist/` artifacts.
